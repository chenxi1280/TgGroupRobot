# TgGroupRobot 功能重构文档

## 1. 重构目标
当前项目存在功能不完整、模块耦合高、配置与行为不一致、异常处理不统一等问题。本次重构目标：

1. 基于现有业务菜单，完整重建“可用、可测、可维护”的功能体系。
2. 所有功能做到“界面配置可见即生效”，并支持明确的开关状态与默认行为。
3. 统一数据模型、回调协议、状态机流程、任务调度机制，降低历史兼容导致的隐性故障。
4. 建立端到端测试与回归清单，确保每项重构可验收、可发布。

---

## 2. 重构范围
### 2.1 在范围内
1. 管理菜单全部功能模块（含截图中所有入口）。
2. 群管理核心链路：配置、检测、处罚、消息反馈、权限校验。
3. 定时任务系统（广告、定时消息、验证超时、清理任务等）。
4. 数据库结构、初始化脚本、迁移兼容策略。
5. 交互回调协议、会话状态、异常提示与日志规范。

### 2.2 暂不在范围
1. 非业务必要的 UI 美化。
2. 与现有业务无关的全新第三方平台接入。
3. 一次性大规模历史数据清洗（除兼容性必要修复外）。

---

## 3. 重构原则
1. **先稳定后扩展**：先修复结构性问题，再补功能细节。
2. **先主链路后边缘**：优先处理影响群管控和消息发送主流程的模块。
3. **配置驱动**：所有行为必须由配置明确控制，并有默认值。
4. **兼容优先**：旧数据、旧回调、旧菜单入口尽量平滑兼容。
5. **可测试**：每个模块必须有最小可运行单测 + 回归清单。
6. **按钮一致性**：管理端按钮文案、顺序、分组必须与确认截图一致；如需改版必须先更新本文档。

---

## 4. 总体重构阶段

### 阶段 A：基线与骨架（先做）
1. 统一模块边界（Handler / Service / Repository / Task）。
2. 统一 callback_data 协议与解析规则。
3. 统一状态机与会话存储规范（私聊/群聊上下文隔离）。
4. 补齐数据库初始化与兼容迁移策略（`init.sql` + 增量兼容脚本）。

### 阶段 B：核心功能逐项重构（按你给的截图顺序推进）
1. 每个功能独立拆解：功能定义 → 数据结构 → 交互流程 → 异常场景 → 测试用例。
2. 完成一项即进入可验收状态，不等全部功能做完再联调。

### 阶段 C：联调与发布
1. 全链路回归（私聊管理、群内执行、定时任务、权限边界）。
2. 压测/长稳（任务调度与数据库连接恢复能力）。
3. 发布清单与回滚策略。

## 10. 功能设计（已展开）

## 轮播广告
### 当前实现状态
- 当前状态：`部分实现`
- 现有入口：私聊主菜单 `轮播广告`；回调 `adm:menu:ads:{chat_id}`；广告模块 `ads:*`；命令 `/ad`。
- 源码证据点：`bot/handlers/ads_handler.py` `bot/services/automation/ad_service.py` `bot/tasks/ads_task.py` `tests/test_ads_config_and_schedule.py`
- 已实现能力：已有轮播广告基础版菜单、创建、列表、看板、启停、定频推送、发送锁、下一次发送时间和次数进度展示。
- 未实现能力：缺少轮播规则页、游标推进、删上轮、发送并置顶，以及独立轮播状态机。
- 排斥/错配：文档目标是独立轮播系统，现状仍是复用广告表 + 广告任务的基础版实现，不具备规则页、游标和删上轮等完整能力。

### 10.1 功能目标
1. 支持在一个群内维护多条“轮播消息”，按固定间隔循环发送。
2. 支持发送模式切换：仅发送 / 发送并置顶。
3. 支持删除与置顶策略配置，避免群内消息堆积或置顶冲突。
4. 支持单条消息启用/禁用、顺序调整、过期清理。

### 10.2 当前问题（来自截图与现状）
1. 规则配置与消息管理分离，但缺少统一状态模型，容易出现“配置显示正常但执行不一致”。
2. 轮播规则较多（删除规则、取消置顶、间隔、开始时间），缺少明确优先级定义。
3. 历史时间字段展示完整，但对“未到开始时间”“已过期”“无可发消息”的处理路径未标准化。
4. 置顶与删除联动场景复杂，容易出现误删或未取消置顶。

### 10.3 目标行为（验收标准）
1. 规则页展示字段与真实执行一致，修改后立即生效。
2. 到达 `next_run_at` 时只发送 1 条“当前顺序且启用且未过期”的消息，然后正确推进到下一条顺序。
3. 删除规则严格生效：
   1. 不删：不删除任何旧轮播消息。
   2. 删上条：发送当前条前，删除“上一条已发送的消息 ID”。
   3. 删上轮：删除“当前内容在上一轮的消息 ID”（同顺序回环点）。
   4. 延迟删：按配置延迟删除当前发送消息。
4. 发送+置顶模式下，可按开关取消上一条轮播置顶。
5. 消息管理支持：新增、修改、删除、启用/关闭、顺序调整、过期清理。

### 10.4 数据模型与配置项
#### A. 轮播规则（每群唯一）
- 表：`ad_rotation_rules`（新建，或复用现有广告配置表并扩展）
- 主字段：
  1. `chat_id`（PK/FK）
  2. `enabled`（轮播状态）
  3. `start_at`（开始时间，UTC 时间戳）
  4. `interval_seconds`（轮播间隔，最小建议 600 秒）
  5. `mode`（`send` / `send_pin`）
  6. `delete_policy`（`none` / `delete_prev` / `delete_prev_cycle` / `delete_delay`）
  7. `delete_delay_seconds`（仅 `delete_delay` 时有效）
  8. `unpin_previous`（取消上一条轮播置顶）
  9. `last_sent_at`
  10. `next_run_at`
  11. `current_order_cursor`（当前轮播游标）
  12. `updated_at`

#### B. 轮播内容（每群多条）
- 表：`ad_rotation_items`
- 主字段：
  1. `id`（PK）
  2. `chat_id`（FK）
  3. `title`
  4. `content_text`
  5. `media_type`（none/photo/video/document/animation）
  6. `media_file_id`（仅存 Telegram file_id）
  7. `buttons`（JSON）
  8. `sort_order`（顺序）
  9. `enabled`
  10. `end_at`（可空，空=永久）
  11. `last_sent_at`
  12. `last_sent_message_id`
  13. `last_sent_cycle_no`（用于“删上轮”）
  14. `created_at/updated_at`

#### C. 默认值
1. `enabled=false`
2. `interval_seconds=7200`（2小时）
3. `mode=send`
4. `delete_policy=delete_prev_cycle`
5. `unpin_previous=true`

#### D. 兼容策略
1. 若历史广告表已存在数据，提供一次迁移脚本将“可轮播广告”迁入 `ad_rotation_items`。
2. 对旧 callback 格式保留兼容路由，统一落到新 service。

### 10.6 服务与任务设计
#### A. Handler
1. `ad_rotation_handler.py`：仅处理菜单渲染、回调分发、状态机输入。

#### B. Service
1. `AdRotationRuleService`：规则读写、参数校验、next_run 计算。
2. `AdRotationItemService`：消息 CRUD、顺序维护、过期清理。
3. `AdRotationDispatchService`：发送、删除、置顶、取消置顶、日志落库。

#### C. Task
1. `ad_rotation_task` 每 60 秒扫描 `enabled=true 且 next_run_at<=now` 的群。
2. 群级加锁（`FOR UPDATE SKIP LOCKED`）避免重复发送。
3. 每次只推进一个游标，发送完成后更新 `current_order_cursor + next_run_at`。

### 10.7 权限与风控
1. 仅群管理员可配置轮播规则与内容。
2. 机器人需具备 `delete/pin` 权限；无权限时降级为“仅发送并记录告警”。
3. 发送失败按错误类型处理：
   1. 可重试错误：保留游标，短期重试。
   2. 不可重试错误（消息失效、媒体失效）：禁用该条并通知管理员。
4. 防止重复发送：同一群同一轮播点位使用幂等键（chat_id + run_slot + item_id）。

### 10.8 测试清单
#### 单测
1. `next_run_at` 计算（含 start_at 未到、跨天、间隔边界）。
2. 顺序游标推进（1>2>3>1）。
3. 四种删除策略行为。
4. 发送模式（send/send_pin）与取消置顶行为。
5. 过期清理与启用过滤。

#### 集成测试
1. 管理页面改配置后，task 在下一周期按新规则执行。
2. 媒体消息发送成功后回写 `last_sent_message_id`。
3. 删除/置顶失败的降级与告警路径。

#### 回归测试
1. 与原“广告管理”模块并存不冲突。
2. 重启后游标与 next_run_at 不丢失。
3. 多群并行轮播互不干扰。

### 10.9 发布与回滚
1. 发布步骤：
   1. 执行新表/新索引 SQL。
   2. 部署代码，默认关闭轮播总开关。
   3. 小流量群灰度开启并观察 24 小时。

3. `scheduled_message_task.py`：调度消费与发送执行器。

#### B. 调度消费流程（核心）
1. 拉取：`enabled=true AND next_run_at <= now`，按 `next_run_at ASC`，`FOR UPDATE SKIP LOCKED`。
2. 消费每条任务：
   1. 若已过 `end_at`：置 `enabled=false` 并跳过发送。
   2. 发送消息（文本/媒体/按钮组合）。
   3. 记录 `last_sent_message_id`。
   4. 计算并更新 `next_run_at = now + repeat_interval_min * 60`（或基于上次计划时间推进，避免漂移）。
3. 提交事务，继续下一条任务。

#### C. 去重与幂等
1. 同一任务单次轮询只允许一次成功发送。
2. 发送失败不更新 `next_run_at` 时需有重试上限/退避，避免疯狂重试。

### 15.1 功能目标
1. 提供群内统一积分体系：签到得分、发言得分、邀请得分、转让积分、抽奖消耗积分。
2. 支持管理员手工加减积分指令，带备注审计。
3. 支持积分查询与排行查询（积分排行、发言排行）。
4. 支持规则可配置：分值、每日上限、最小字数、展示开关。

### 15.2 当前问题（来自截图与现状）
1. 主页面入口多且联动复杂（签到/发言/邀请/转让/抽奖/规则/日志），缺少统一配置中心描述。

2. 签到规则：
   1. 用户发送“签到”每日仅获得一次签到积分。
   2. 可配置签到文案与积分数量，支持签到人数显示开关。
3. 发言规则：
   1. 满足最小字数才计分。
   2. 每日获取上限生效，超限不再加分。
4. 邀请规则：
   1. 邀请入群成功后加分。
   2. 每日上限生效。
5. 管理指令：
   1. 回复用户发送 `加积分 数字 备注` 成功加分并写审计日志。
   2. 回复用户发送 `扣积分 数字 备注` 成功扣分并写审计日志。

### 15.6 服务与执行设计

3. 执行账本变更并写 `points_ledger`。

   1. 出价不得超过用户积分余额。

## 车库转发
### 当前实现状态
- 当前状态：`部分实现`
- 现有入口：私聊主菜单 `车库转发`；回调 `adm:menu:garage_forward:{chat_id}`、`gfw:*`。
- 源码证据点：`bot/handlers/admin_handler.py` `bot/services/integration/garage_forward_service.py`。
- 已实现能力：已有来源绑定、关键字规则、转发模式与映射表主链路；新增转发审计日志页（全部/成功/跳过/失败筛选）与首页审计统计摘要。
- 未实现能力：仍需补重试队列与更完整的失败回放工具。
- 排斥/错配：基础运营工作台已可用，但距离“完整转发平台”仍有差距。

### 38.1 功能目标
1. 自动同步其他车库频道消息到当前车库，降低车库被炸后内容断档风险。
2. 在车库保安后台统一配置转发来源与规则。
3. 保证同步稳定、可追踪、可防循环。

### 38.2 当前问题（来自截图与需求）
1. 当前界面仅提示“去车库频道后台配置”，缺少标准化配置模型。
2. 频道同步是高频链路，需解决重复转发、循环转发和失败重试问题。
3. 需要明确权限：仅车库频道后台管理员可配置。

### 38.3 目标行为（验收标准）
1. 支持配置多个来源车库频道，自动同步新消息。
2. 支持来源过滤策略（全部/仅文本/仅图文/按关键词）。
3. 同步消息保留来源标识（来源频道、原消息链接或ID映射）。
4. 同一消息只同步一次，且不会被反向再次抓取（防循环）。
5. 同步失败可重试并有失败审计。

### 38.4 数据模型与配置项
#### A. 群级配置（建议 `garage_forward_settings`）
1. `chat_id`（PK，目标车库频道/群）
2. `enabled`
3. `sync_mode`（all/text/media/keyword）
4. `keyword_rules`（jsonb，可空）
5. `updated_at`

#### B. 来源绑定（建议 `garage_forward_sources`）
1. `id`
2. `chat_id`（目标车库）
3. `source_channel_id`
4. `source_name`
5. `enabled`
6. `created_at`

#### C. 消息映射（建议 `garage_forward_message_map`）
1. `id`
2. `chat_id`（目标）
3. `source_channel_id`
4. `source_message_id`
5. `target_message_id`
6. `forwarded_at`
7. 唯一约束：`(chat_id, source_channel_id, source_message_id)`

#### D. 同步审计（建议 `garage_forward_audit_logs`）
1. `id`
2. `chat_id`
3. `source_channel_id`
4. `source_message_id`
5. `action`（forward/retry/skip）
6. `result`
7. `reason`
8. `created_at`

### 38.5 交互与回调协议
1. 回调前缀：`gfw`
2. 示例：
   1. `gfw:home:{chat_id}`
   2. `gfw:toggle:{chat_id}:{0|1}`
   3. `gfw:source:add:{chat_id}`
   4. `gfw:source:del:{chat_id}:{source_id}`
   5. `gfw:mode:set:{chat_id}:{mode}`
3. 输入状态机：
   1. `GFW_WAIT_SOURCE_CHANNEL_INPUT`
   2. `GFW_WAIT_KEYWORD_RULE_INPUT`

### 38.6 服务与执行设计
1. `garage_forward_handler.py`：后台配置页与来源管理。
2. `garage_forward_service.py`：
   1. 消息过滤判定。
   2. 同步执行与映射落库。
   3. 重试队列管理。
3. `garage_forward_task.py`（可选）：
   1. 扫描失败任务并重试。

### 38.7 权限与风控
1. 仅目标车库后台管理员可配置来源。
2. 来源频道需校验机器人具备读取权限。
3. 防循环策略：
   1. 跳过由本机器人发送的消息。
   2. 使用消息映射去重。
4. 防刷保护：单来源频道限速与异常熔断。

### 38.8 测试清单
1. 单测：过滤模式判定、去重键生成、防循环判定。
2. 集成：来源频道发消息后目标车库自动同步成功。
3. 异常：来源无权限、消息删除、网络超时时重试与日志正确。
4. 回归：不影响其他模块（自动回复、轮播、验证）消息链路。
