## 1. 产品定位

TgGroupRobot 是面向 Telegram 群运营场景的群组管理机器人。系统以“一个 Bot 实例管理多个 Telegram 群组”为核心模型，支持每个群独立配置、独立数据、独立权限和独立运营流程。

管理员主要在私聊 Bot 中发送 `/start` 进入按钮化管理面板，通过按钮完成群组切换、功能配置、内容发布、活动管理和日常巡检。普通群成员主要在群内完成新人验证、签到、积分查询、邀请、活动参与、老师搜索、附近搜索、报告提交和车评查看。

当前版本默认开放群组功能。订阅、续费和卡密底座保留，但不作为群功能运行时限制。

## 2. 产品目标

1. 降低 Telegram 群运营门槛，让群主不用记忆复杂命令即可完成配置。
2. 用统一群消息处理链承接风控、积分、邀请、活动和车库业务，避免功能互相绕开。
3. 让多群运营可复制：配置、权限、积分、活动、车库资料均按群隔离，并支持导入或克隆。
4. 形成完整运营闭环：拉新、验证、治理、活跃、积分、活动、自动发布、老师搜索、车评沉淀。
5. 将 Web 后台限定为平台公共能力，不让它承担单群管理配置，避免 Bot 端和后台端职责混乱。

## 3. 系统边界

### 3.1 Bot 端职责

Bot 是主要业务入口，负责所有群级配置和群内运行时业务：

- 多群管理、群组切换、管理员权限校验。
- 群组健康检查、命令启停、命令别名、自动删除系统提示。
- 新人验证、验证超时、入群刷号保护、强制订阅、新成员限制。
- 反垃圾、防刷屏、违禁词、处罚动作降级、惩罚策略同步。
- 签到积分、发言积分、邀请积分、自定义积分、积分等级、积分商城、积分任务。
- 自动回复、定时消息、轮播广告、全局快捷发布、底部按钮。
- 邀请链接、用户自助邀请、邀请归因、邀请排行。
- 抽奖、接龙、游戏、竞猜、促活、拍卖。
- 欢迎消息、改名监控、联盟、联合封禁、夜间管控。
- 车库认证、老师搜索、附近搜索、频道同步、车评系统。

### 3.2 Web 后台职责

Web 后台只负责平台公共能力：

- 平台管理员登录和会话。
- 平台公告配置。
- 续费卡密批量生成、查询、复制、导出、作废。
- 后台账号、后台会话和审计记录。
- 平台公共配置。

Web 后台不负责：

- 不管理单个群的新人验证、风控、积分、邀请、活动、车库、车评。
- 不作为群运营人员的日常工作台。
- 不替代 Bot 私聊管理面板。

## 4. 用户角色

| 角色 | 使用入口 | 核心诉求 | 可用能力 |
|------|----------|----------|----------|
| 平台管理员 | Web 后台 | 管理平台公共资源 | 登录、公告、卡密、账号、审计、公共配置 |
| 群主 | Bot 私聊 `/start` | 管理自己群的完整运营能力 | 群配置、权限、风控、积分、邀请、活动、车库、车评、自动化 |
| 群管理员 | Bot 私聊 `/start` | 在授权范围内协助运营 | 按群权限进入可用模块，操作前再次校验权限 |
| 报告审核人员 | Bot 通知与私聊按钮 | 审核老师报告 | 查看报告、通过、驳回、触发车评生成 |
| 普通群成员 | 群内消息和按钮 | 参与群运营玩法 | 验证、签到、查积分、邀请、活动、老师搜索、报告提交、车评查询 |
| 车库老师 | 群内发言和私聊输入 | 维护老师身份与位置 | 认证展示、位置更新、开课打卡、被搜索、被评价 |

## 5. 总体架构要求

当前代码采用运行时装配、平台能力、功能域和共享能力分层。

```text
main.py
  -> backend.app.runtime.main()
    -> build_application()
      -> 加载 Settings
      -> 创建 Database(engine + session_factory)
      -> 构建 python-telegram-bot Application
      -> 注入 settings/db
      -> 注册命令、功能 Router、通用消息处理器、错误处理器
    -> 启动迁移 + schema gate
    -> app initialize/start
    -> Telegram polling
    -> Scheduler.start()
    -> 可选启动 FastAPI Web 后台
```

目录职责：

| 目录 | 职责 |
|------|------|
| `backend/app` | 应用装配、启动链路、Router 注册、消息分发 |
| `backend/platform` | 配置、数据库、状态、Telegram 适配、调度器 |
| `backend/features` | 业务功能域，按 vertical slice 组织 |
| `backend/shared` | 权限、状态、发布、UI、callback 解析等跨域能力 |
| `docs-site` | 功能手册、流程图、文档校验和构建 |
| `docs` | 产品、架构、部署、功能基线文档 |

功能 Router 由 `backend/app/router_registry.py` 统一注册，当前包括：

- `AdminRouter`
- `LotteryRouter`
- `SolitaireRouter`
- `InviteRouter`
- `AdsRouter`
- `ScheduledMessageRouter`
- `AutoReplyRouter`
- `BannedWordRouter`
- `PointsRouter`
- `RenewalRouter`
- `NearbyRouter`
- `GroupRouter`
- `VerificationRouter`
- `BottomButtonRouter`
- `GameRuntimeRouter`

新增功能必须优先进入独立 Router 或既有功能域 Router，不应继续把所有逻辑堆进启动文件。

## 6. 总体交互流程

### 6.1 管理员配置流程

```text
管理员私聊 Bot 发送 /start
-> Bot 查询该用户可管理群组
-> 管理员选择目标群
-> 系统校验 Telegram 管理员权限
-> 展示目标群管理面板
-> 管理员进入功能模块
-> 按钮或文本输入修改配置
-> 系统保存到目标 chat_id
-> 群内运行时按新配置生效
```

流程要求：

- 所有群级配置必须绑定目标 `chat_id`。
- 私聊输入状态必须能定位到当前被管理群，避免文本输入写错群。
- 敏感操作需要在进入或提交时校验 Telegram 管理权限。
- 返回按钮必须回到当前模块或当前群上下文，不得串到其他群或其他模块。

### 6.2 群消息处理流程

```text
群内产生消息或成员事件
-> 原始 update 日志探针
-> 系统消息自动删除
-> 防刷屏
-> 反垃圾 / 链接 / 黑名单 / 违禁词
-> 命令别名
-> 统一 MessageDispatcher
-> GroupMessageHandler
-> 群 hooks 前置规则
-> 积分 / 邀请 / 活动 / 自动回复 / 车库业务
-> 返回群内响应或静默处理
```

流程要求：

- 风控优先于普通业务处理。
- 媒体、caption、系统事件也要进入可适配的处理链。
- 无法确认邀请来源时不得误发邀请积分。
- 机器人权限不足时，处罚动作必须降级到可执行动作并给出可理解提示。

### 6.3 私聊输入流程

```text
管理员点击配置按钮
-> 系统写入 private/config state
-> 管理员发送文本、媒体、位置或 /clear
-> MessageDispatcher 解析当前 state 和 managed_chat_id
-> PrivateConfigHandler 分发到对应输入处理器
-> 校验格式
-> 保存配置
-> 清理或更新 state
-> 重新渲染功能页
```

流程要求：

- 输入页必须说明格式、示例和清空方式。
- `/clear` 应能清理当前输入状态。
- 输入失败不得破坏已有配置。
- 媒体输入、位置输入、按钮输入和时间输入必须走状态机，不依赖群内命令。

### 6.4 Web 后台流程

```text
平台管理员打开 Web 后台
-> 输入账号密码
-> 校验账号状态和密码
-> 创建后台会话
-> 进入后台首页
-> 管理卡密 / 公告 / 账号 / 配置
-> 记录审计日志
-> 退出或会话过期
```

流程要求：

- Web 后台不进入单群配置。
- 禁用账号、重置密码或修改密码后，应使旧会话失效。
- 卡密导出、复制、作废等操作需留下审计记录。

## 8. 关键功能需求

### 8.1 群组核心

#### 多群隔离与私聊切群

需求：

- 一个 Bot 实例可以管理多个群。
- 每个群的配置、积分、邀请、活动、风控、车库数据必须按 `chat_id` 隔离。
- 管理员在私聊中选择当前管理群，后续按钮和输入都落到该群。
- callback data 或状态中应显式携带目标群，不能依赖模糊上下文。

验收：

- 管理员在 A 群和 B 群之间切换后，同一功能配置互不覆盖。
- 私聊输入状态能正确回写目标群。
- 从任意子页面返回时仍停留在当前群上下文。

#### 管理员权限校验

需求：

- 管理入口必须校验 Telegram 群管理员身份。
- 敏感操作提交时再次校验权限。
- 权限不足时只提示无权限，不执行数据变更。

验收：

- 非管理员无法进入群管理面板。
- 管理员被移除后，再次操作会被拦截。
- 权限不足不会留下半写入配置。

#### 群组健康检查

需求：

- 集中展示机器人权限、验证、强制订阅、反垃圾、防刷屏、关群、定时消息等运行状态。
- 风险项应给出可跳转修复入口。

验收：

- 机器人缺少删消息、禁言、封禁等权限时可被识别。
- 管理员能从检查页跳到对应配置页。

### 8.2 新人与风控

#### 新人验证

需求：

- 支持按钮、数学题、验证码、管理员审核。
- 新成员入群后根据配置决定是否限制发言。
- 验证成功后解除限制。
- 验证消息应具备超时处理。

流程：

```text
新成员入群
-> 系统读取群验证配置
-> 按模式发送验证消息
-> 成员点击按钮或提交答案
-> 校验通过
-> 解除限制并清理挑战
```

异常：

- 超时未完成：禁言、踢出或按配置处理。
- 机器人权限不足：提示管理员修复权限。
- 重复点击：提示已处理，不重复执行解除或处罚。

#### 反垃圾、防刷屏和违禁词

需求：

- 反垃圾覆盖链接、广告、转发、超长内容、黑名单等规则。
- 防刷屏按频率阈值触发删除、禁言或封禁。
- 违禁词按群维护，支持启停、增删改查和处罚动作。
- 风控处罚需要支持权限不足降级。

流程：

```text
群消息进入
-> 防刷屏检测
-> 反垃圾检测
-> 违禁词检测
-> 命中规则
-> 计算处罚动作
-> 检查机器人权限
-> 执行删除 / 警告 / 禁言 / 封禁 / 降级动作
-> 记录结果
```

验收：

- 触发风控时不应继续进入会造成奖励的业务逻辑。
- 权限不足时不抛出未处理异常。
- 命中记录可用于统计或复盘。

### 8.3 积分运营

需求：

- 支持签到、发言、邀请三类基础积分来源。
- 支持主积分中心、发言排行、个人发言量、转让、管理员加减分、日志导出、清空。
- 支持自定义积分类型、等级、商城和任务页。
- 积分变更必须有流水，不能只改余额。

流程：

```text
用户触发签到 / 发言 / 邀请
-> 检查功能开关和频率限制
-> 计算积分
-> 写入积分流水
-> 更新积分账户余额
-> 返回结果或更新排行
```

验收：

- 同一用户同一群积分独立。
- 签到冷却和连续签到奖励生效。
- 邀请积分只在可靠归因存在时发放。
- 商城订单能从兑换到处理形成闭环。

### 8.4 消息自动化

需求：

- 自动回复支持关键词、封面、按钮、排序和延迟删除。
- 定时消息支持文本、媒体、按钮、开始/结束时间、重复间隔、详情和删除。
- 定时消息按钮输入同时支持 `文本|链接` 逐行格式和 JSON。
- 轮播广告支持单次推送、定时开始、间隔轮播、图片配置和看板。
- 快捷发布支持私聊一键投放文本、媒体和按钮。
- 底部按钮支持文本、布局、事件绑定、自定义触发词和同步到底部键盘。

验收：

- 配置保存后可预览或通过测试消息验证。
- 定时任务在调度器中执行，不阻塞 polling。
- 按钮格式错误时提示格式，不破坏原配置。
- 底部按钮配置流程必须包含底部文本、按钮布局和同步三步。

### 8.5 邀请增长

需求：

- 管理员可创建、查看、统计、预览、清零、清空和导出邀请链接。
- 普通用户可自助生成邀请链接。
- 邀请归因必须基于可靠 `ChatMemberUpdate` invite metadata。
- 邀请排行展示前 10 和成员数据。

验收：

- 无法确认来源时不得误发邀请积分。
- 邀请统计、排行和导出数据一致。
- 用户自助邀请入口可被管理员启停或配置。

### 8.6 活动互动

需求：

- 抽奖支持创建、参与条件、手动或自动开奖。
- 接龙支持创建、加入和状态跟踪。
- 游戏支持玩法设置、牌局列表、定时启停和运行时动作。
- 竞猜支持活动设置、选项、庄家、奖池、截止时间和结算。
- 促活工具支持彩蛋活动、奖励配置、统计和领奖记录。
- 拍卖支持设置、活动列表、出价和结算。

验收：

- 每类活动有管理入口、用户参与入口和结束/结算流程。
- 活动状态从创建到结束可追踪。
- 中奖、奖励、结算等关键结果不得重复执行。

### 8.7 车库、老师搜索与车评

需求：

- 车库认证支持启停、认证图标、手动添加/删除认证老师、联盟共享认证池。
- 老师发言限制支持关闭、仅图片、图文模式、时间间隔、限制条数和白名单。
- 老师搜索支持标签、附近搜索、出勤状态、底部按钮。
- 附近搜索需要用户或老师更新位置。
- 车评系统支持报告提交、审核、发布目标、自定义项、报告列表、审核日志和排行。
- 报告审核人员可以收到通知并处理通过或驳回。

车评流程：

```text
用户从回复老师消息 / 搜索结果 / 命令入口提交报告
-> 系统校验老师档案和提交条件
-> 按报告模式收集内容
-> 推送审核人员
-> 审核通过
-> 生成或更新车评
-> 群内可查询、排行、展示
```

异常：

- 老师未录入档案：提示无法提交或引导录入。
- 老师停用：不展示或禁止提交。
- 没有审核人员：提示管理员配置。
- 审核人员权限失效：不能继续审核。
- 重复审核：提示已处理。

### 8.8 频道管理与发布

需求：

- 频道同步支持来源、模式、关键词过滤和同步日志。
- 频道自动按钮支持同步时附加按钮模板。
- 修改按钮能力面向最近或指定可编辑同步消息。
- 全局快捷发布支持文本、媒体和按钮一键投放。

验收：

- 频道同步和车库转发复用能力时，页面文案必须说明名称关系。
- 机器人无权读取来源或发送目标时提示修复权限。
- 同步日志可用于排查消息是否被过滤。

### 8.9 Web 后台公共功能

需求：

- 后台账号支持登录、会话和状态校验。
- 卡密支持批量生成、复制、导出、作废、使用状态查看。
- 公告支持创建、编辑、下线和前端展示。
- 平台公共配置可集中维护。
- 操作日志记录关键后台动作。

验收：

- 禁用账号、重置密码、改密码后旧会话失效。
- 卡密导出文件可打开且数据与查询一致。
- Web 后台不出现单群风控、积分、活动、车库配置入口。

## 9. 状态设计

| 对象 | 主要状态 | 说明 |
|------|----------|------|
| 群 | 未绑定、可管理、权限不足、功能开启、功能关闭 | 群级配置和权限检查的基础 |
| 成员 | 新成员、验证中、已验证、受限、被处罚、普通成员、管理员 | 决定验证、风控和权限 |
| 私聊输入 | 无状态、选中群、等待文本、等待媒体、等待位置、等待按钮、等待时间 | 决定 PrivateConfigHandler 路由 |
| 邀请 | 已创建、已使用、已清零、已导出、已失效 | 邀请统计和奖励依据 |
| 积分 | 可用余额、冻结或待处理订单、流水已记录 | 积分账户必须由流水支撑 |
| 活动 | 草稿、进行中、已截止、已开奖、已结算、已取消 | 防止重复开奖或重复结算 |
| 老师 | 已认证、未认证、停用、位置已更新、今日开课 | 决定搜索、展示和报告入口 |
| 报告 | 待审核、已通过、已驳回、已生成车评 | 车评沉淀链路 |
| 卡密 | 未使用、已绑定、已作废、已过期 | Web 后台公共能力 |

## 10. 数据与实现约束

1. 群级数据必须优先以 `chat_id` 隔离。
2. 业务变更应写入对应 service，Handler 不承载复杂业务规则。
3. 私聊配置必须使用状态服务记录上下文，不用临时全局变量。
4. 调度任务必须进入统一 scheduler，不在 handler 中阻塞等待。
5. 数据库启动前必须经过启动迁移和 schema gate。
6. docs-site 的 `catalog.json`、`flows/*.json`、功能 MDX 和真值表必须保持一致。
7. 新增功能需同步：代码入口、功能真值表、docs-site catalog、flow、测试。

## 11. 非功能需求

### 11.1 稳定性

- Telegram polling、调度器和 Web 后台应能独立启动和优雅退出。
- 单个调度任务失败不得拖垮全部任务。
- Telegram API 异常需要可记录、可降级、可恢复。

### 11.2 可维护性

- 新功能按功能域 vertical slice 组织。
- 公共能力沉淀到 `shared` 或 `platform`，不得跨域复制大段逻辑。
- 回调数据应具备可解析前缀和目标群上下文。

### 11.3 可观测性

- 原始 update、消息分发、风控命中、调度任务和 Web 后台关键操作需要结构化日志。
- 错误提示面向用户时应简洁，面向日志时应保留上下文。

### 11.4 安全性

- 管理员权限必须来自 Telegram 当前权限或系统明确授权。
- Web 后台会话、密码、卡密不能在日志中明文泄露。
- 平台后台不能越权操作单群配置。

## 13. 当前范围与后续演进

当前版本范围：

- 群组核心、入群风控、内容审核、积分运营、消息自动化、邀请增长、群组扩展、活动互动、频道管理均以已闭环为目标。
- Web 后台保持平台公共功能定位。
- 会员 / 套餐授权暂时关闭，只保留订阅和续费底座。

后续演进建议：

1. 将 PRD 和 docs-site catalog 建立自动对账，避免功能新增后 PRD 漏更新。
2. 将关键业务流程沉淀为端到端验收清单，覆盖按钮返回、私聊输入、群消息触发和调度任务。
3. 对增长型数据表补充归档策略，例如积分流水、违规记录、邀请事件和活动日志。
4. 对高频群消息链路增加性能基线，明确单群高频消息下的数据库和 Telegram API 调用上限。
5. 会员 / 套餐授权恢复前，需要重新定义运行时 gating 规则，避免影响当前默认开放策略。

## 15. 通知与消息规则

### 15.1 管理员通知

触发场景：

- 群健康检查发现机器人权限不足。
- 新人验证、强制订阅、风控等关键配置无法执行。
- 定时消息、轮播广告、活动开奖、拍卖结算等后台任务失败。
- 报告审核人员缺失或权限失效。
- Web 后台关键账号或卡密操作完成。

消息要求：

- 管理员通知应说明问题、影响范围和修复入口。
- 群级问题必须包含目标群信息。
- 不向群内公开后台配置细节、卡密、内部审计字段和敏感错误堆栈。

### 15.2 用户通知

触发场景：

- 新人验证通过、失败、超时。
- 强制订阅未完成。
- 风控处罚、处罚降级或消息被删除。
- 签到、积分变更、商城兑换、邀请奖励。
- 抽奖、接龙、竞猜、拍卖等活动参与结果。
- 老师搜索、附近搜索、报告提交、车评查询。

消息要求：

- 用户通知优先使用可执行按钮或下一步提示。
- 失败提示应告诉用户如何恢复，例如重新验证、补充位置、换关键词搜索。
- 群内公开消息不得展示管理后台专用字段。

### 15.3 老师通知

触发场景：

- 老师认证状态变更。
- 强制更新位置。
- 开课状态变更。
- 老师被提交报告且业务需要提示。
- 车评发布或排行变化。

消息要求：

- 老师通知应避免暴露提交人的隐私信息。
- 位置更新流程必须说明可提交的位置格式。
- 老师停用或未认证时，应给出联系管理员或重新认证路径。

## 16. 异常与降级策略

| 场景 | 系统行为 | 用户可见提示 | 数据要求 |
|------|----------|--------------|----------|
| 机器人缺少删消息权限 | 不执行删除，尝试警告或记录 | 提示管理员补权限 | 记录权限不足原因 |
| 机器人缺少禁言权限 | 处罚降级为删除或警告 | 提示处罚已降级 | 记录原动作和实际动作 |
| Telegram API 请求失败 | 捕获异常并记录日志 | 普通用户只看到简洁失败提示 | 保留 update、chat、user、action 上下文 |
| 数据库不可用 | 启动期阻断，运行期记录并返回失败 | 提示稍后重试 | 不产生半写入 |
| 私聊状态丢失 | 回到功能入口或当前群管理页 | 提示重新选择功能 | 不复用过期输入 |
| 输入格式错误 | 不保存新值 | 展示格式和示例 | 保留原配置 |
| 重复点击按钮 | 返回已处理或当前状态 | 提示无需重复操作 | 幂等处理 |
| 邀请来源不可靠 | 不发邀请奖励 | 可提示无法确认邀请来源 | 不写奖励流水 |
| 活动重复开奖 / 结算 | 阻止重复执行 | 提示活动已处理 | 保留首次结果 |
| 报告重复审核 | 阻止重复审核 | 提示报告已处理 | 保留审核日志 |
| 审核人员权限失效 | 阻止审核操作 | 提示联系群主重新配置 | 记录权限失效 |
| 频道同步来源不可读 | 不同步该消息 | 提示检查来源权限 | 写同步失败日志 |
| Web 会话过期 | 要求重新登录 | 跳转登录 | 清理失效会话 |

## 17. 模块级验收清单

### 17.1 群组核心验收

- `/start` 私聊入口能展示可管理群。
- 切换群后所有配置页都带目标群上下文。
- 群健康检查能识别至少机器人权限、验证、强制订阅、反垃圾、防刷屏、关群、定时消息。
- 自动删除能处理进群、退群、置顶、头像、群名、匿名消息。
- 导入设置和克隆必须在确认后覆盖目标模块，不得误覆盖未选模块。

### 17.2 入群风控验收

- 新成员入群能触发配置的验证模式。
- 验证通过能解除限制，验证超时能执行配置动作。
- 强制订阅在未完成订阅时能拦截发言。
- 新成员限制能按入群时长限制媒体、链接和纯文本。
- 风控命中后不会继续执行积分奖励等后置业务。

### 17.3 内容审核验收

- 违禁词支持新增、启停、删除、命中处罚。
- 反垃圾支持链接、广告、转发、超长内容和黑名单规则。
- 防刷屏支持阈值、窗口、处罚动作和禁言时长。
- 权限不足时处罚降级，不抛出未处理异常。
- 惩罚策略页能同步多个风控模块的处罚动作。

### 17.4 积分运营验收

- 签到、发言、邀请积分都能写余额和流水。
- 管理员加减分、积分转让、清空和导出结果一致。
- 自定义积分类型之间余额互不混淆。
- 积分等级按门槛展示。
- 积分商城从商品配置、兑换、订单处理到库存变化形成闭环。

### 17.5 消息自动化验收

- 自动回复命中关键词后返回配置内容、封面和按钮。
- 定时消息支持文本、媒体、按钮和重复计划。
- 定时消息详情页能删除或查看任务。
- 轮播广告能按间隔推送，失败时不阻塞其他调度任务。
- 快捷发布支持文本、媒体、按钮一键投放。
- 底部按钮完成“底部文本 -> 按钮布局 -> 同步到底部键盘”三步后可在群内触发。

### 17.6 邀请增长验收

- 管理员邀请链接可以创建、预览、统计、清零、导出。
- 普通用户邀请链接可以自助生成。
- 新成员通过可靠 invite metadata 入群后能归因。
- 无法确认来源时不发奖励。
- 邀请排行与统计数据一致。

### 17.7 活动互动验收

- 抽奖能创建、限制参与条件、参与、手动或自动开奖。
- 接龙能创建、加入、查看状态。
- 游戏能配置玩法并查看牌局。
- 竞猜能创建选项、参与、截止和结算。
- 促活工具能配置奖励并查看统计。
- 拍卖能创建拍品、出价和结算。

### 17.8 车库与车评验收

- 车库认证能启停、配置认证图标、增删认证老师。
- 老师发言限制能按模式和频率生效。
- 老师搜索支持关键词、附近、开课老师等入口。
- 老师或用户位置更新后，附近搜索能按距离返回。
- 报告提交支持回复老师消息、搜索结果和命令入口。
- 报告审核通过后能生成车评，驳回后不会公开展示。
- 车评排行和查询只使用已审核或已发布数据。

### 17.9 频道管理与发布验收

- 频道同步能配置来源、模式、关键词过滤和日志。
- 按钮模板能附加到同步消息。
- 修改按钮只作用于机器人可编辑的消息。
- 快捷发布能在目标群或频道完成投放。

### 17.10 Web 后台验收

- 平台管理员能登录和退出。
- 禁用账号、重置密码、修改密码后旧会话失效。
- 卡密批量生成、复制、导出、作废和状态查看可用。
- 公告能创建、编辑、下线并在前端展示。
- Web 后台不出现单群功能配置入口。

## 21. 详细功能流程（按设计 PRD 展开）

本章是产品评审、开发补齐和测试验收的主口径。前文的功能清单、追溯矩阵和证据索引用于确认覆盖范围，本章负责说明每个功能到底是什么、谁使用、如何配置、如何触发、异常如何处理。

### 21.1 系统入口

#### 管理员私聊入口

功能说明：

- 管理员通过私聊 Bot 发送 `/start` 进入管理面板。
- 私聊面板是群组配置、运营管理、活动发布、车库维护和快捷发布的主入口。
- 管理面板需要围绕当前群组展示，不应要求管理员记忆 `/admin` 之类的额外命令。

入口内容：

- 当前群组。
- 切换群组。
- 群组体检。
- 基础设置。
- 新人验证。
- 风控审核。
- 积分中心。
- 邀请增长。
- 自动消息。
- 活动互动。
- 车库运营。
- 群组扩展。
- 平台公告查看。

流程：

```text
管理员私聊发送 /start
-> 系统识别 Telegram 用户
-> 查询该用户可管理的群组
-> 如果没有可管理群组
    -> 提示先将机器人加入群并设为管理员
-> 如果有可管理群组
    -> 展示群组列表
-> 管理员选择群组
-> 系统校验管理员权限
-> 进入该群管理首页
```

验收要点：

- 管理员看见的是按钮流程，不是命令列表。
- 多群管理员必须先选群或使用上次选中的群。
- 权限失效时不能继续进入群配置。

#### 群内用户入口

功能说明：

- 普通群成员主要通过群内关键词、命令、底部按钮、活动按钮使用功能。
- 群内入口先经过风控和状态检查，再进入具体业务功能。

常用入口：

```text
签到
积分
排行
邀请
邀请排行
任务
老师搜索 关键词
附近
附近 关键词
开课老师
车评 老师名
车评排行
报告 老师名 内容
```

流程：

```text
用户在群内发送消息
-> 系统先执行群状态检查
-> 执行新人验证检查

```

验收要点：

- 风控未通过的消息不能触发积分、活动、自动回复。
- 命令关闭或别名未命中时，应按配置提示或静默。
- 用户入口提示必须有下一步，而不是只返回“失败”。

- 登录。

- 公告管理。


### 21.2 群内实时处理总流程

功能说明：

- 群内所有消息、成员变化、按钮点击都进入统一处理链。
- 系统先治理风险，再处理运营功能。
- 如果消息被删除、禁言、踢出，则不再触发积分、活动、自动回复等后续业务。

总流程：

```text
群内产生消息
-> 判断是否来自机器人自己
    -> 是：忽略
    -> 否：继续
-> 判断群是否已接入
    -> 未接入：忽略
    -> 已接入：继续
-> 判断群状态
    -> 暂停服务：忽略或提示
    -> 夜间管控：执行夜间规则
    -> 正常：继续
-> 判断是否为系统消息
    -> 是：执行系统消息自动删除
    -> 否：继续
-> 判断用户状态
    -> 未验证：执行新人验证拦截
    -> 黑名单：执行黑名单规则
    -> 正常：继续
-> 执行强制订阅检查
-> 执行新成员限制检查
-> 执行违禁词检查
-> 执行反垃圾检查
-> 执行防刷屏检查
-> 执行命令识别
-> 执行业务功能
    -> 签到 / 积分 / 邀请
    -> 活动参与
    -> 老师搜索 / 附近搜索
    -> 报告提交 / 车评查询
-> 执行发言积分奖励
-> 执行自动回复
-> 记录处理结果
```

规则：

- 群状态和用户状态是业务处理前置条件。
- 风控处理优先于运营奖励。
- 运行时必须兼容文本、媒体、caption、系统事件。
- 命中严重规则后应终止后续业务。

独立配置包括：

- 积分规则。
- 邀请规则。


验收要点：

- A 群配置不影响 B 群。
- 私聊输入状态必须携带目标群。

校验场景：

```text
管理员发起操作

```

- 违禁词是否配置。
- 定时消息是否异常。
- 车库是否开启。
- 老师档案是否存在。
- 报告审核人员是否配置。
- 夜间管控是否冲突。

流程：

```text
管理员点击“群组体检”

-> 系统检查机器人权限

-> 给出修复建议
```

#### 群命令启停与别名

功能说明：

- 管理员可以控制群内命令是否启用，并为命令设置别名。

```text
管理员进入“命令管理”
```

### 21.4 新人与风控功能

#### 新人验证

功能说明：

- 失败处理方式。
- 验证提示文案。
- 验证成功文案。
- 验证失败文案。
- 是否删除验证消息。
- 是否仅对新人启用。

管理员配置流程：

```text
管理员进入“新人验证”
-> 点击开启新人验证
-> 选择验证方式
-> 设置超时时间
-> 设置失败处理方式
    -> 禁言
    -> 踢出
    -> 加入观察名单
-> 设置验证提示文案

```

新人入群流程：

```text
新人加入群组
-> 系统判断新人验证是否开启
-> 未开启
    -> 进入欢迎消息流程
-> 已开启
    -> 限制新人发言
    -> 发送验证提示
    -> 等待新人完成验证
```

验证通过流程：

```text
新人完成验证
-> 系统校验验证结果

```

验证超时流程：

```text
新人进入待验证状态
-> 系统开始计时
-> 到达超时时间
-> 判断是否已验证
-> 未验证
    -> 按配置执行禁言、踢出或观察
    -> 通知管理员或记录日志
```

#### 入群刷号保护

功能说明：

- 用于识别短时间大量账号入群、低质量账号集中入群、同一邀请链接异常增长等风险。

检测维度：

- 短时间入群人数过多。
- 同一邀请链接入群过多。
- 多个无头像账号入群。
- 用户名相似账号集中入群。
- 入群后立即发链接。
- 入群后立即发媒体。
- 入群后立即重复发言。

流程：

```text
新成员加入群
-> 系统记录入群事件
-> 判断当前时间窗口内入群数量
-> 判断邀请链接是否异常
-> 判断账号特征
-> 计算风险等级
-> 低风险
    -> 正常进入验证或欢迎流程
-> 中风险
    -> 加强验证或限制发言
-> 高风险
    -> 禁言、踢出或通知管理员
```

#### 强制订阅

功能说明：

- 要求用户先加入指定频道或群组，才能在当前群正常发言或使用功能。

管理员配置流程：

```text
管理员进入“强制订阅”
-> 添加需要订阅的频道或群
-> 设置提示文案
-> 设置按钮
-> 设置未订阅处理方式
-> 开启强制订阅
```

用户流程：

```text
用户发言或使用功能
-> 系统检查用户是否已订阅目标频道或群
-> 已订阅
    -> 正常使用
-> 未订阅
    -> 删除或拦截消息
    -> 发送订阅提示
    -> 用户点击加入
    -> 用户点击“我已完成”
    -> 系统复查
    -> 通过后恢复使用
```

#### 新成员限制

功能说明：

- 控制新人入群后一段时间内的行为，降低广告和刷屏风险。

可限制：

- 图片、视频、文件、链接、纯文本、转发消息、表情包、高频发言。

流程：

```text
新成员发送消息
-> 系统判断入群时长
-> 已超过限制期
    -> 正常处理
-> 仍在限制期
    -> 判断消息类型是否被限制
-> 未命中限制
    -> 继续后续流程
-> 命中限制
    -> 删除消息
    -> 发送提示
    -> 记录违规次数
-> 超过违规次数
    -> 禁言或踢出
```

#### 反垃圾

功能说明：

- 识别广告、诈骗、诱导点击、异常联系方式、恶意推广等内容。

检测内容：

- 广告链接。
- 邀请链接。
- 可疑联系方式。
- 大量特殊符号。
- 重复内容。
- 诈骗关键词。
- 黑产关键词。
- 高频转发。
- 可疑媒体。
- 新号异常行为。

流程：

```text
用户发送消息
-> 系统提取文本、链接、媒体、转发来源
-> 匹配反垃圾规则
-> 未命中
    -> 进入下一流程
-> 命中
    -> 计算风险等级
-> 低风险
    -> 提醒或删除
-> 中风险
    -> 删除并禁言
-> 高风险
    -> 删除、踢出或加入黑名单
-> 记录风控日志
```

#### 防刷屏

功能说明：

- 限制用户短时间大量发言、重复发言、连续发图片或表情包。

流程：

```text
用户发送消息
-> 系统统计该用户在时间窗口内的消息数量
-> 判断是否超过阈值
-> 未超过
    -> 正常通过
-> 超过
    -> 判断历史违规次数
-> 第一次超限
    -> 删除消息并提醒
-> 第二次超限
    -> 删除消息并短时禁言
-> 多次超限
    -> 长时间禁言或踢出
```

#### 违禁词

功能说明：

- 拦截群内敏感词、广告词、诈骗词、自定义禁用词。

支持匹配方式：

管理员配置流程：

```text

-> 选择匹配方式
-> 选择违禁词等级

-> 保存规则
```

用户触发流程：

```text
用户发送消息
-> 系统提取文本
-> 匹配违禁词库
-> 未命中
    -> 正常处理
-> 命中
    -> 判断等级
    -> 执行删除、警告、禁言或踢出
    -> 记录命中日志
```

#### 黑名单

功能说明：

- 禁止特定用户、链接、关键词或来源在群内活动。

黑名单对象：

- 用户 ID。

- 链接域名。

- 关键词。
- 频道来源。
- 媒体指纹。

```text
用户入群或发言
```

#### 处罚动作降级

功能说明：

### 21.5 积分与会员运营功能

#### 发言积分

- 用户正常发言可获得积分，用于提升群活跃度。

支持配置：

- 每条消息奖励积分、最低字数、每日奖励上限、冷却时间、媒体消息是否计分、表情包是否计分。

流程：

```text
用户发送普通消息
-> 系统先执行风控
-> 风控通过
    -> 判断发言积分是否开启
-> 判断消息是否满足字数和类型要求
-> 判断是否处于冷却时间
-> 判断今日奖励是否达到上限
-> 满足条件
    -> 发放发言积分
-> 不满足条件
    -> 不发积分
```

不发积分的情况：

- 消息被删除。
- 命中违禁词。
- 触发防刷屏。
- 用户未完成新人验证。
- 用户在黑名单中。
- 达到每日奖励上限。
- 处于冷却时间。

### 21.9 车库、老师搜索、报告与车评功能

#### 模块定位

功能说明：

- 车库模块是一个由管理员维护的老师档案库，不是老师自助资料系统。
- 老师搜索基于管理员维护的车库资料。
- 报告是用户提交的一条待审核内容。
- 车评是报告审核通过后生成的正式评价。

完整闭环：

```text
管理员录入老师档案
-> 管理员设置认证和展示规则
-> 老师只更新位置
-> 用户搜索老师
-> 用户提交报告
-> Bot 推送给指定审核人员
-> 审核人员通过或不通过
-> 通过后生成正式车评
-> 车评进入老师详情、查询、排行
```

#### 车库总开关

功能说明：

- 控制整个车库模块是否启用。

关闭后不可用：

- 老师搜索、附近老师、开课老师、老师详情、报告提交、车评查询、车评排行。
- 历史老师资料、报告、车评不删除。

流程：

```text
管理员进入“车库运营”
-> 查看车库当前状态
-> 点击开启或关闭
-> 系统检查必要配置
    -> 是否已有老师档案
    -> 是否配置审核人员
    -> 是否配置报告规则
-> 配置完整
    -> 车库开启成功
-> 配置不完整
    -> 提示补充配置
```

功能说明：

- 老师档案全部由管理员录入、编辑、停用。
- 老师本人不能修改档案。

老师档案包括：

- 展示名称、Telegram 用户、用户名、认证图标、地区、地址描述、价格说明、服务标签、简介、联系方式展示规则、是否认证、是否进入搜索、是否进入附近搜索、是否展示车评、排序权重、管理员备注。

添加老师流程：

```text

-> 选择添加方式

    -> 输入用户名
    -> 从群成员中选择
-> 系统识别目标用户
-> 管理员填写老师资料
-> 设置认证状态
-> 设置是否进入搜索
-> 设置是否允许提交报告
-> 管理员确认保存
-> 系统创建老师档案
-> 可选通知老师本人
```


#### 报告功能

功能说明：

- 报告是用户对某位老师提交的待审核内容。
- 报告不是最终车评。
- 报告必须经过指定审核人员审核，通过后才成为正式老师车评。
- 当前用户提交报告有两个真实入口：
  - 群内回复老师消息后发送提交指令。
  - 群内发送“提交报告”或点击底部按钮“提交车评”，再跳转到 Bot 私聊分步提交。
- 从老师搜索结果提交报告属于产品目标入口；如果搜索结果页提供“提交报告”按钮，该按钮应落到同一个 Bot 私聊提交流程。

入口前置条件：

- 车评系统已开启。
- 管理员已配置审核人。
- 目标老师是当前群有效认证老师。
- 群内“报告/车评”命令未被命令配置关闭。
- 用户消息已通过新人验证、强制订阅、新成员限制和风控链路。

报告状态：

- 待审核、已通过、未通过、已撤回、已删除。
- 基础版本至少支持：待审核 -> 通过 / 不通过。

提交入口一：群内回复老师消息提交。

适用场景：

- 老师刚在群内发言，用户可以直接回复老师消息提交体验报告。
- 系统通过被回复消息的 `from_user` 识别目标老师。

用户操作：

```text
1. 用户在群内找到目标老师的一条消息。
2. 用户点击 Telegram 的“回复”。
3. 用户输入：提交报告 + 报告内容。
   示例：
   提交报告 人照：5
   颜值：5
   身材：4
   服务：5
   态度：5
   环境：4
   综合：5
   评价：整体体验不错
   过程：按约定到达，沟通顺畅
4. 用户发送消息。
```

系统流程：

```text
用户回复老师消息
-> 用户发送“提交报告 + 报告内容”
-> 系统检查车评系统是否开启
-> 系统检查报告/车评命令是否启用
-> 系统检查是否配置审核人
-> 系统读取被回复消息的发送人
-> 判断被回复对象是否为当前群有效老师
-> 不是有效老师
    -> 提示“提交车评请回复目标老师的消息后再发送指令”或提示老师无效
-> 是有效老师
    -> 解析报告内容
    -> 默认模式校验必填评分项
    -> 简易模式只校验正文是否可用
    -> 提取随消息附带的图片/媒体 file_id
    -> 创建 car_review_reports
    -> report_status = pending
    -> 推送审核通知给审核人
    -> 群内回复提交人：已提交车评报告，等待审核。报告ID：xxx
```

失败提示：

- 未回复老师消息：提示“提交车评请回复目标老师的消息后再发送指令。”
- 未配置审核人：提示“车评系统还没有配置审核人，请管理员先在车评系统里指定审核人。”
- 指令被关闭：提示“该指令已关闭。”
- 默认模式缺少字段：提示缺少哪些字段。
- 分数格式错误：提示具体字段分数格式不正确。

提交入口二：群内进入私聊分步提交。

适用场景：

- 用户不知道要回复哪条老师消息。
- 用户通过底部按钮“提交车评”进入。
- 用户在群内只发送“提交报告”或“提交车评”，希望系统引导提交。

用户操作：

```text
1. 用户在群内发送“提交报告”或点击底部按钮“提交车评”。
2. Bot 在群内回复说明，并展示“去私聊提交车评”按钮。
3. 用户点击按钮跳转到 Bot 私聊。
4. Bot 私聊提示目标群，并要求先发送老师 @用户名 或 Telegram ID。
5. 用户发送老师 @用户名 或 Telegram ID。
6. Bot 校验该用户是否为当前群有效认证老师。
7. 校验通过后，Bot 提示用户发送车评内容。
8. 用户按模板发送文字，也可以附带图片。
9. Bot 创建待审核报告，并通知审核人。
```

系统流程：

```text
群内收到“提交报告”或“提交车评”
-> 如果没有回复老师消息
-> 检查车评系统和审核人
-> 生成私聊深链 /start crvsub_{chat_id}
-> 群内提示用户点击“去私聊提交车评”
-> 用户进入私聊
-> start_car_review_submit_flow 检查目标群是否存在
-> 检查车评系统是否开启
-> 检查审核人是否存在
-> 写 conversation_states
    -> state_type = car_review_submit_teacher_input
    -> state_data.target_chat_id = 目标群
-> 用户发送老师 @用户名 或 Telegram ID
-> 系统解析 TgUser
-> 检查是否为当前群有效认证老师
-> 不是老师
    -> 提示重新发送老师 @用户名 或 Telegram ID
-> 是老师
    -> 写 conversation_states
        -> state_type = car_review_submit_body_input
        -> state_data.target_chat_id = 目标群
        -> state_data.teacher_user_id = 老师用户 ID
-> 用户发送评价内容和可选图片
-> 系统解析模板字段、评分和正文
-> 创建 car_review_reports
-> 清理 conversation_states
-> 私聊回复：车评已提交，等待审核。报告ID：xxx
-> 私聊审核人并附带“查看报告 / 通过 / 驳回”按钮
```

失败提示：

- 私聊入口已失效：提示用户回群重新点击“提交车评”。
- 目标群不存在：提示入口已失效。
- 车评系统未开启：提示“该群暂未启用车评系统。”
- 未配置审核人：提示先联系管理员配置。
- 老师输入无法解析：提示发送老师 @用户名 或 Telegram ID。
- 老师不是当前群有效认证老师：提示重新发送老师。
- 评价内容缺少默认模式字段：提示缺少字段。

提交入口三：从老师搜索结果提交。

产品目标：

- 用户搜索老师后，应能从老师列表或老师详情直接点击“提交报告 / 提交车评”。
- 该按钮不应让用户重新猜老师身份，而应携带老师上下文进入提交流程。

推荐流程：

```text
用户发送“老师搜索 关键词”
-> 系统返回老师列表
-> 每个老师结果展示“查看详情”和“提交报告”
-> 用户点击某老师的“提交报告”
-> 系统生成私聊提交入口
-> 私聊 state_data 同时携带 target_chat_id 和 teacher_user_id
-> Bot 私聊直接进入“发送车评内容”步骤
-> 用户按模板发送评价内容和图片
-> 系统创建待审核报告
-> 通知审核人
```

实现约束：

- 如果按钮已携带 `teacher_user_id`，私聊流程可以跳过“输入老师”步骤。
- 如果按钮只携带群上下文，则仍走“先输入老师”步骤。
- 老师在点击后被停用或取消认证时，提交时必须重新校验并提示不可提交。

提交入口四：命令带老师关键词提交。

产品目标：

- 支持用户直接发送“报告 老师A 内容”或“提交报告 老师A 内容”。
- 当系统无法唯一识别老师时，不能直接创建报告。

推荐流程：

```text
用户发送“报告 老师A 内容”
-> 系统识别老师关键词
-> 匹配老师档案
-> 未匹配到老师
    -> 提示未找到老师，并建议使用“老师搜索 关键词”
-> 匹配到多个老师
    -> 返回候选老师按钮，要求用户选择
-> 匹配到唯一老师
    -> 校验该老师是否允许提交报告
    -> 解析报告内容
    -> 创建待审核报告
    -> 通知审核人
```

当前实现说明：

- 当前可靠主路径是“回复老师消息 + 提交报告内容”和“群内触发后进入私聊分步提交”。
- “老师搜索结果直接提交”和“命令携带老师关键词提交”应作为后续产品化入口，与同一套 `car_review_reports` 数据流复用。

报告内容格式：

默认模式示例：

```text
人照：5
颜值：5
身材：4
服务：5
态度：5
环境：4
综合：5
评价：整体体验不错
过程：按约定到达，沟通顺畅
```

简易模式示例：

```text
整体体验不错，沟通顺畅，环境干净，服务态度好。
```

带图片提交：

```text
用户在群内回复老师消息发送“提交报告 ...”并附图
或在私聊提交正文时附图
-> 系统提取 media_file_ids
-> 保存到 car_review_reports.media_file_ids
-> 审核通过发布时按发布配置带出媒体
```

提交条件：

- 用户必须完成新人验证。
- 用户不能在黑名单中。
- 用户不能被禁言。
- 用户每天最多提交几条报告。
- 用户对同一老师每天最多提交几条报告。
- 用户必须在群内停留超过一定时间。
- 用户必须有最低积分。
- 用户不能是该老师本人。
- 老师必须允许被提交报告。

内容格式：

- 简易模式：用户提交一段文字，系统检查字数后创建报告。
- 标准模式：系统发送模板，用户按模板填写。
- 评分项模式：用户填写评分和评价，系统校验评分是否合法。

报告创建后的数据：

| 字段 | 来源 |
|------|------|
| `chat_id` | 当前群 |
| `teacher_user_id` | 被回复消息发送人、私聊输入老师、或搜索结果老师 |
| `author_user_id` | 提交报告的用户 |
| `review_text` | 用户评价正文 |
| `process_text` | 用户填写的过程内容 |
| `scores` | 模板评分项解析结果 |
| `media_file_ids` | 用户提交时附带的图片或媒体 |
| `report_status` | 初始为 `pending` |

报告提交后的用户反馈：

```text
提交成功
-> 群内或私聊回复：车评已提交，等待审核。报告ID：xxx

提交失败
-> 告诉用户具体原因
-> 保持可重试路径
```

#### 审核人员配置与审核

审核人员可以是：

- 群主。
- 群管理员。
- 指定 Telegram 用户。
- 专门的车评审核员。
- 多个审核人组成的审核组。

配置流程：

```text
管理员进入“报告审核设置”
-> 添加审核人员
    -> 从群管理员中选择
    -> 输入用户 ID
    -> 回复用户消息添加
-> 设置审核通知方式
    -> 私聊通知
    -> 管理群通知
    -> 同时通知
-> 设置审核规则
    -> 任意一人审核即可
    -> 多人审核通过才生效
-> 保存配置
```

报告推送流程：

```text
用户提交报告
-> 系统创建待审核报告
-> 系统查询审核人员
-> 存在审核人员
    -> Bot 私聊推送审核卡片
-> 没有审核人员
    -> 通知群主配置审核人员
    -> 报告保持待审核状态
```

审核通过流程：

```text
审核人员点击“通过”
-> 系统判断审核人员权限
-> 判断报告是否仍是待审核
-> 报告状态改为已通过
-> 生成正式老师车评
-> 更新老师评分和车评数量
-> 判断是否自动发布到群
-> 判断是否奖励提交人积分
-> 通知提交人审核通过
-> 通知其他审核人员该报告已处理
```

审核不通过流程：

```text
审核人员点击“不通过”
-> 系统判断审核人员权限
-> 判断报告是否仍是待审核
-> 系统要求选择或填写不通过原因
-> 报告状态改为未通过
-> 通知提交人审核未通过
-> 通知其他审核人员该报告已处理
-> 记录审核结果
```

#### 报告生成车评

功能说明：

- 车评不是用户直接创建的，而是报告审核通过后生成的正式评价。

流程：

```text
报告审核通过
-> 系统读取报告内容
-> 生成正式车评
-> 绑定对应老师
-> 计算评分
-> 更新老师平均评分
-> 更新老师车评数量
-> 判断是否发布到群
-> 判断是否展示在老师详情
```

#### 车评展示、查询和排行

展示规则：

- 只展示审核通过的车评。
- 不展示未通过报告。
- 提交人默认匿名。
- 可配置是否展示评分、时间、完整内容、摘要、低评分车评、是否参与排行。

车评查询流程：

```text
用户发送“车评 老师A”
-> 系统识别老师关键词
-> 查询老师档案
-> 未找到
    -> 提示未找到老师
-> 找到多个
    -> 提示用户选择
-> 找到唯一
    -> 查询该老师已通过车评
    -> 按时间、评分或热度排序
    -> 返回车评摘要
```

车评排行流程：

```text
用户发送“车评排行”
-> 系统判断排行周期
    -> 总排行
    -> 本周排行
    -> 本月排行
-> 查询已通过车评
-> 统计老师评分和评价数量
-> 生成排行榜
-> 群内展示
```

报告与车评积分奖励：

```text
报告审核通过
-> 系统判断是否开启报告奖励
-> 开启
    -> 给提交人发放积分
    -> 通知提交人奖励结果
-> 未开启
    -> 不发积分
```

车库异常流程：

- 老师未录入档案：不进入报告流程，可提示该用户不是已录入老师。
- 老师已停用：拒绝提交，提示该老师当前不可提交报告。
- 没有审核人员：通知群主配置审核人员，报告保持待审核。
- 审核人员权限失效：拒绝操作，提示无审核权限。
- 重复审核：提示该报告已处理，不改变结果。

### 21.15 一期业务闭环

一期完成后，系统应形成以下完整闭环：

```text
群接入
-> 管理员私聊 Bot 配置群功能
-> 新人入群验证
-> 群内风控治理
-> 用户签到和发言获得积分
-> 用户邀请新人获得积分
-> 管理员发布定时消息和抽奖
-> 管理员录入老师档案
-> 老师只更新位置
-> 用户搜索老师或附近老师
-> 用户提交老师报告
-> Bot 推送给指定审核人员
-> 审核人员通过或不通过
-> 通过后生成正式车评
-> 车评进入老师详情、车评查询、车评排行
-> Web 后台只处理卡密、公告、公共配置和日志
```

重要规则：

- Web 后台只做平台公共功能，群相关操作全部在 Bot 私聊面板完成。
- 老师资料归管理员维护，老师本人不能修改资料，只能更新位置。
- 报告是用户提交的待审核内容，审核通过后才生成正式车评。
- 审核不通过则不展示、不计分、不进排行。
- 每个群配置独立。
- 群内风控优先于积分和活动。
- 用户命中风控后不发放积分、不参与活动、不触发自动回复。
- 所有关键操作需要权限校验。

## 22. 功能点击路径与用户使用流程

本章说明每个功能在 Bot 私聊管理面板中的点击路径、普通用户在群内或私聊中的使用路径，以及流程完成后应产生的数据或状态变化。

通用点击约定：

```text
管理员私聊 Bot
-> 发送 /start
-> 选择当前群或使用已选群
-> 进入管理首页
-> 点击一级模块
-> 点击二级功能
-> 通过按钮、文本输入、媒体输入、位置输入完成配置
-> 保存后返回当前功能页或管理首页
```

通用用户约定：

```text
用户在群内发送关键词 / 命令 / 点击按钮
-> 系统先执行群状态、用户状态、强制订阅、新人限制、风控检查
-> 通过后进入具体业务
-> 业务处理完成后返回群内消息、私聊通知或静默记录
```

| 功能 | 管理员点击路径 | 用户使用路径 | 数据 / 状态变化 |
|------|----------------|--------------|-----------------|
| 多群隔离 / 私聊切群 | `/start` -> 切换群 -> 选择目标群 -> 进入该群管理首页 | 无普通用户路径 | `conversation_states` 记录当前管理群；后续 callback 和输入绑定目标 `chat_id` |
| 管理员权限校验 | `/start` -> 任意管理模块 -> 系统自动校验 | 非管理员点击管理入口时被拒绝 | 不写业务数据；记录权限不足日志 |
| 群组健康检查 | `/start` -> 健康检查 -> 查看异常项 -> 点击修复入口 | 无普通用户路径 | 读取 Bot 权限和群配置，不直接改变配置 |
| 自动删除 | `/start` -> 删除提示 -> 选择删除类型 -> 开启/关闭 -> 保存 | 群内出现系统提示消息 | 更新 `chat_settings` 自动删除配置；命中后删除 Telegram 系统消息 |
| 控制权限 | `/start` -> 管理权限 -> 选择策略 -> 保存 | 权限不足用户操作时被拦截 | 更新群级权限策略；后续管理操作按策略判断 |
| 群组统计 | `/start` -> 群组统计 -> 查看成员/邀请/积分/违规摘要 | 无普通用户路径 | 读取统计表和业务表，不直接改变状态 |
| 新人验证 | `/start` -> 进群验证 -> 选择验证模式 -> 设置超时/文案/失败动作 -> 保存 | 新人入群 -> 点击按钮或提交答案 -> 通过/失败 | 创建或更新 `verification_challenges`；成员状态从未验证到已验证或被处罚 |
| 验证超时处理 | `/start` -> 进群验证 -> 验证超时 -> 设置处理动作和时间 -> 保存 | 新人未验证等待超时 | `verification_challenges` 到期；调度任务执行禁言/踢出/提示 |
| 入群刷号保护 | `/start` -> 垃圾防护 -> 入群规则 -> 设置阈值/动作 -> 保存 | 新成员集中入群或入群后异常发言 | 写入风险记录或风控日志；可能限制、禁言、踢出 |
| 强制订阅 | `/start` -> 强制关注 -> 添加频道/群 -> 设置提示和按钮 -> 开启 | 用户发言 -> 点击订阅按钮 -> 点击我已完成 -> 复查 | 更新强制订阅配置；用户未订阅时消息被删除或拦截 |

## 23. 数据流转与状态迁移

本章说明关键业务对象从创建、使用、更新到结束的状态流转。表名以当前代码模型为依据，字段级细节以数据库模型和迁移为准。

### 23.1 群接入与群配置数据流

涉及对象：

- `tg_chats`
- `chat_members`
- `chat_settings`
- `conversation_states`

流转：

```text
Bot 被加入群
-> 收到群事件或管理员私聊 /start
-> 识别 chat_id、群标题、群类型
-> 写入或更新 tg_chats
-> 识别管理员和成员关系
-> 写入或更新 chat_members
-> 初始化 chat_settings 默认配置
-> 管理员私聊选择群
-> conversation_states 记录 selected_chat / managed_chat_id
-> 后续所有配置写入该 chat_id
```

状态说明：

| 状态 | 含义 | 下一步 |
|------|------|--------|
| 未接入 | Bot 未识别或未保存该群 | Bot 加入群并触发初始化 |
| 已接入 | 群记录存在 | 校验 Bot 权限 |
| 权限不足 | Bot 不是管理员或缺少关键权限 | 管理员补权限 |
| 可管理 | Bot 权限正常，管理员权限正常 | 进入功能配置 |
| 已退出 / 不可用 | Bot 被移出或群不可访问 | 停止运行时处理 |

### 23.2 私聊配置输入状态流

涉及对象：

- `conversation_states`
- 各业务配置表。

流转：

```text
管理员点击需要输入的按钮
-> 写入 conversation_states
    -> chat_id = 目标群
    -> user_id = 管理员
    -> state_type = 当前输入类型
    -> state_data = 业务上下文
-> 管理员发送文本 / 媒体 / 位置 / /clear
-> MessageDispatcher 读取私聊 state
-> PrivateConfigHandler 分发到业务输入处理器
-> 输入合法
    -> 写业务配置表
    -> 清理或更新 conversation_states
    -> 重新渲染功能页
-> 输入非法
    -> 保留原配置
    -> 返回格式示例
```

状态迁移：

```text
无状态
-> 等待文本 / 等待媒体 / 等待位置 / 等待按钮 / 等待时间
-> 输入成功
-> 已保存并清理状态

等待输入
-> /clear
-> 清理状态

等待输入
-> 输入非法
-> 保持等待输入
```

### 23.3 新人验证数据流

涉及对象：

- `verification_challenges`
- `chat_members`
- `chat_settings`
- Telegram 成员权限。

流转：

```text
新成员入群
-> 读取 chat_settings 验证配置
-> 创建 verification_challenges
    -> user_id
    -> chat_id
    -> token / 答案
    -> expires_at
    -> solved = false
-> 限制新成员发言
-> 用户完成验证
-> 校验 token / 答案 / 审核按钮
-> solved = true
-> 解除发言限制
-> 更新成员状态
```

状态迁移：

```text
未验证
-> 待验证
-> 验证通过
-> 已验证

待验证
-> 超时
-> 禁言 / 踢出 / 观察

待验证
-> 验证失败
-> 重试或处罚
```

### 23.4 风控与处罚数据流

涉及对象：

- `banned_words`
- `moderation_violations`
- `moderation_warnings`
- 反垃圾和防刷屏运行时计数。

流转：

```text

-> 提取文本、caption、链接、媒体、转发来源
-> 匹配违禁词 / 反垃圾 / 防刷屏规则
-> 未命中
    -> 进入业务流程
-> 命中
    -> 写 moderation_violations
    -> 计算处罚动作
    -> 检查 Bot 权限
    -> 执行原动作或降级动作
    -> 必要时写 moderation_warnings
    -> 终止后续业务
```

处罚状态：

```text
待处理
-> 已删除消息
-> 已警告
-> 已禁言
-> 已踢出
-> 已封禁
```

## 24. 端到端点击验收脚本

本章提供可手动执行的端到端验收流，覆盖管理员点击、用户使用、数据状态和异常处理。

### 24.1 新群接入和基础配置

```text
1. 将 Bot 加入测试群，并设为管理员。
2. 管理员私聊 Bot 发送 /start。
3. 点击“切换群”，选择测试群。
4. 点击“健康检查”，确认 Bot 权限。
5. 点击“进群验证”，开启按钮验证，设置超时和失败动作。
6. 点击“垃圾防护”，开启链接检测和黑名单规则。
7. 点击“防刷屏”，设置短时间消息阈值。
8. 点击“主积分”，设置签到积分和发言积分。
9. 点击“进群欢迎”，设置欢迎文案。
10. 邀请一个新用户进群，检查验证、欢迎、风控和积分是否按顺序生效。
```

验收数据：

- `tg_chats` 有测试群。
- `chat_settings` 有该群配置。
- 新人验证时有 `verification_challenges`。
- 签到或发言后有 `points_transactions`。

### 24.2 风控阻断积分和自动回复

```text
1. 管理员在“违禁词”中添加测试词。
2. 管理员在“自动回复”中添加同样关键词的回复规则。
3. 普通用户在群内发送包含测试词的消息。
4. 系统应先命中违禁词，删除或处罚。
5. 系统不应发放发言积分。
6. 系统不应触发自动回复。
```

验收数据：

- `moderation_violations` 写入命中记录。
- `points_transactions` 不新增该消息的发言积分流水。
- 群内不出现自动回复内容。

### 24.3 积分、邀请和排行闭环

```text
7. 用户发送“邀请排行”，查看用户 A 排名变化。
```

验收数据：

- `invite_links` 有用户 A 的专属链接。
- `invite_tracking` 记录用户 B 的归因。
- `points_transactions` 有邀请积分流水。
- 排行数据与有效邀请统计一致。

### 24.4 定时消息和按钮输入闭环

### 24.6 车库老师搜索、报告和车评闭环

```text

11. 普通用户回复老师消息，并发送：
    提交报告 人照：5
    颜值：5
    身材：4

    环境：4
```

## 25. 全功能细节核查矩阵

本章用于逐项检查所有功能是否写清楚“点击路径、用户使用、数据流转、状态变化、异常处理”。后续如果新增功能，必须在本章补一行，避免 PRD 只写“支持某功能”但没有落地流程。

| 功能 | 管理员点击细节 | 用户使用细节 | 数据对象与状态流转 | 异常与边界 |
|------|----------------|--------------|--------------------|------------|
| 多群隔离 / 私聊切群 | `/start` 后进入群列表，点击目标群，系统回到该群管理首页。 | 普通成员无入口；影响所有群内功能的目标群归属。 | `tg_chats` 保存群，`chat_members` 保存成员，`conversation_states` 保存 `managed_chat_id`。 | 群不存在、Bot 已退出、管理员权限失效时拒绝切换。 |
| 管理员权限校验 | 进入任意管理模块或提交敏感操作前自动校验。 | 非管理员点击管理入口时只收到无权限提示。 | 不写业务数据；只读取 Telegram 管理员状态和群权限策略。 | 管理员被移除后，已打开页面的后续提交也必须失败。 |

| 控制权限 | `/start` -> 管理权限，设置哪些管理能力可由管理员操作。 | 受限管理员尝试操作时被拦截。 | 写群级权限策略，后续功能提交前读取。 | 权限策略不能绕过 Telegram 原生管理员校验。 |
| 群组统计 | `/start` -> 群组统计，查看成员、邀请、积分、违规摘要。 | 普通成员通常无入口；可通过排行榜看到部分统计。 | 读取 `chat_members`、积分表、邀请表、违规表和每日统计。 | 数据为空时展示空状态，不报错。 |
| 新人验证 | `/start` -> 进群验证，选择按钮、数学题、验证码或人工审核，设置超时和失败动作。 | 新人入群后点击按钮或提交答案，通过后解除限制。 | 创建 `verification_challenges`，状态从待验证到 solved 或超时处罚。 | 答案错误、重复点击、超时、Bot 无禁言权限均需提示或降级。 |
| 验证超时处理 | `/start` -> 进群验证 -> 验证超时，设置时间和禁言或踢出动作。 | 新人未验证时等待倒计时，到期后被处理。 | 调度任务扫描 `verification_challenges.expires_at`，执行处罚并更新状态。 | 用户已验证则跳过；权限不足则降级并通知管理员。 |
| 入群刷号保护 | `/start` -> 垃圾防护 -> 入群规则，设置时间窗口、人数阈值和处理动作。 | 新成员集中入群时被加强验证、限制或踢出。 | 记录入群事件和风险命中，必要时写违规记录。 | 同一邀请链接异常增长要能单独提示管理员。 |
| 强制订阅 | `/start` -> 强制关注，添加频道或群，配置封面、按钮和未订阅处理动作。 | 用户发言时未订阅会收到加入按钮，点击完成后复查。 | 写强制订阅配置；运行时读取用户订阅状态。 | Bot 无法读取目标频道成员时提示配置不可用。 |
| 新成员限制 | `/start` -> 新成员限制，设置限制时长、消息类型和违规次数动作。 | 新成员在限制期内发链接、媒体或文本时被删除或提示。 | 读取成员入群时间和 `chat_settings` 限制项，命中后写违规记录。 | 超过限制期自动放行；管理员和白名单需豁免。 |
| 违禁词 | `/start` -> 违禁词，新增词条、匹配方式、分类和处罚动作。 | 用户发送命中内容时消息被删除、警告、禁言或踢出。 | 写 `banned_words`；命中后写 `moderation_violations`。 | 正则错误应在保存时拦截；权限不足走处罚降级。 |
| 反垃圾 | `/start` -> 垃圾防护，配置链接、广告、转发、超长内容和黑名单规则。 | 用户发送广告、邀请链接或可疑媒体时被处理。 | 写反垃圾配置；运行时写违规记录和警告记录。 | 白名单用户、管理员、已关闭规则要跳过。 |
| 防刷屏 | `/start` -> 防刷屏，设置窗口、阈值、禁言时长和清理消息。 | 用户短时间刷消息时被提醒、删除、禁言。 | 运行时维护频率计数，命中后写违规记录。 | 计数窗口过期后恢复；权限不足时降级。 |
| 处罚动作降级 | 在风控动作选择处配置目标动作。 | 用户触发风控时根据 Bot 权限执行实际动作。 | 违规记录保存原动作和实际动作。 | 踢出不可用降级禁言，禁言不可用降级删除，删除不可用通知管理员。 |
