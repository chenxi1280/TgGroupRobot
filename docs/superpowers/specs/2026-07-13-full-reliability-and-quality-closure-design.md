## 1. 目标

本设计覆盖 2026-07-13 代码审计确认的全部实现、功能设计、操作、数据流与工程质量问题。完成态不是“测试仍然通过”，而是以下事实同时成立：

1. 验证超时、车库转发、定时消息和轮播广告具备明确、可恢复、可审计的持久化状态。
2. Telegram 外部副作用不会因为数据库收尾失败而被系统自动重复执行。
3. 明确失败可自动重试，结果不确定时停止自动动作并进入人工处理队列。
4. 管理员可查看失败、永久失败和不确定任务，并能显式重放或关闭。
5. 定时消息、广告规则和历史能力与 PRD 描述一致，实际菜单与后台可操作。
6. Schema 变更有版本记录，启动校验覆盖关键结构属性。
7. 群消息异常不会再把编程错误伪装为业务校验失败。
8. 审计指出的 10 个超大后端文件和 237 个超长函数全部达到 500/50 行硬限制，且建立持续门禁。
9. 全量回归、静态检查、迁移验证、关键状态机测试全部通过且没有未解释警告。

## 3. 方案比较

### 方案 A：按领域持久化状态，共用执行语义（采用）

各领域保留自己的表和业务字段，共享不可变状态枚举、退避计算、租约规则和执行结果类型。优势是业务语义清晰、迁移范围可控、现有服务可逐步替换；代价是各领域仍有少量显式适配代码。

### 方案 B：建设统一 Telegram Outbox 表（不采用）

所有发送都进入一个通用表，由中央 worker 派发。它能减少重复代码，但会把验证、转发、定时消息和广告的不同完成条件压入通用 payload 与回调注册表，扩大一次性迁移风险，也容易形成新的平台层业务耦合。

### 方案 C：局部补丁并保留现有 claim 语义（不采用）

只增加唯一索引、异常捕获和几个测试，交付快，但无法解决发送后数据库失败、人工重放、历史查询、PRD 不一致和调度健康误报，不满足“全部完成”。

## 4. 统一可靠性语义

### 4.1 状态

所有需要 Telegram 外部副作用的执行记录使用以下状态：

- `pending`：等待首次执行。
- `processing`：已被 worker 认领，租约有效。
- `retryable_failed`：Telegram 明确返回未执行，等待退避重试。
- `succeeded`：Telegram 成功且数据库业务收尾完成。
- `permanent_failed`：超过最大次数或错误明确不可恢复。
- `uncertain`：请求可能已经执行，但系统未能确认或持久化结果；禁止自动重发。
- `cancelled`：管理员显式关闭。

### 4.2 不可变执行结果

平台层提供不可变的 `DeliveryOutcome`：

- `success(message_id, metadata)`
- `retryable_failure(error_code, message)`
- `permanent_failure(error_code, message)`
- `uncertain(error_code, message)`

业务 worker 注入 Telegram 发送器和仓储接口，不直接构造具体数据库或 Bot 实现。

### 4.3 原子性边界

PostgreSQL 与 Telegram API 无法组成真正的跨系统原子事务。本设计采用明确的“有效一次”语义：

1. 数据库先创建执行记录并提交。
2. worker 认领记录，写入 `processing`、租约和 `send_started_at` 后提交。
3. 调用 Telegram。
4. 明确失败进入可重试或永久失败。
5. 成功后写入消息 ID、业务游标、审计与 `succeeded`。
6. 如果调用超时或发送后数据库结果无法确认，记录进入 `uncertain`；租约回收器也只会把有 `send_started_at` 的失联任务转为 `uncertain`，不会自动重发。
7. 管理员确认后才能重放 `uncertain`，每次重放生成新 attempt 并保留原记录。

该规则优先消除系统自动重复发送，同时完整暴露可能未发送的任务，不做静默丢弃。

## 5. 领域设计

### 5.1 验证超时

在 `VerificationChallenge` 增加：

- `timeout_status`
- `timeout_action`
- `timeout_attempts`
- `timeout_next_retry_at`
- `timeout_lease_until`
- `timeout_send_started_at`
- `timeout_last_error`
- `timeout_completed_at`

处理流程：

1. `SELECT ... FOR UPDATE SKIP LOCKED` 认领到期 challenge。
2. 计算并持久化处罚动作，设置租约。
3. 执行 mute/kick/unrestrict。
4. 明确权限或参数错误进入 `permanent_failed`；明确瞬时错误进入 `retryable_failed`；网络结果不确定进入 `uncertain`。
5. 成功后才同步兼容字段 `timeout_handled=True`；`unrestrict` 成功时写 `solved=True`。

管理员验证工作台新增“超时失败任务”，支持查看错误、重试和关闭。旧 `timeout_handled=True` 数据迁移为 `succeeded`，未处理数据迁移为 `pending`。

### 5.2 车库转发

`GarageForwardRetryQueue` 调整为真正的执行记录：

- 对 `(chat_id, source_channel_id, source_message_id)` 建唯一约束。
- 保存 `message_map_id`、按钮 JSON 快照、状态、租约、开始时间、错误、次数和完成时间。
- `enqueue_retry` 使用 PostgreSQL upsert，重复事件只更新同一记录。
- 实时链路的 `GarageForwardMessageMap` 占位在失败后继续保留，不再删除；重试与实时投递共用该占位。
- 重试成功在同一数据库事务内完成 message map、来源游标、审计和队列状态更新。
- worker 对每条任务独立处理，单条收尾失败不终止整批。
- 失联且已开始 Telegram 调用的记录进入 `uncertain`，禁止租约过期后自动复制。

管理员车库工作台新增失败列表、永久失败列表、不确定列表和显式重放动作。

### 5.3 定时消息

`ScheduledMessageLog` 从结果日志升级为每次计划执行的记录，增加：

- `run_key`，唯一值为 `task_id + scheduled_for`。
- `scheduled_for`
- `status`
- `attempt_count`
- `next_retry_at`
- `lease_until`
- `send_started_at`
- `completed_at`
- `error_code`

调度流程：

1. 到期任务只负责创建唯一执行记录，并计算下一次 `next_run_at`。
2. 派发 worker 认领执行记录并发送完整内容快照。
3. 成功、明确失败和不确定分别落库。
4. 调度器 tick 只要存在失败就记录失败健康指标；不能再把 100% 发送失败判定为成功。
5. 管理界面展示成功、重试中、永久失败和不确定历史。

时间窗口改为“计算下一个窗口边界”，不再按重复周期盲目迭代。跨日窗口、12 小时、24 小时、首次开始时间和夏令时无关的 UTC+8 业务时区均有确定测试。

### 5.4 轮播广告

- `select_next_rotation_item` 先排除 `exclude_campaign_ids`，再在非空 `top_campaign_ids` 池中选择；无有效置顶项时返回明确配置错误，不静默回退到全部广告。
- 管理菜单提供置顶池、排除池配置，并校验广告属于当前群。
- `AdRotationHistory` 增加执行状态、attempt、错误、租约、`scheduled_for` 和唯一 `dispatch_key`，发送前创建历史记录。
- 成功后才推进游标和发送计数；明确失败按策略重试；不确定禁止自动重发。
- 管理菜单和 Web 后台提供历史筛选、失败查看和显式重放。

### 5.5 群消息管线

- 新增明确的 `BusinessRuleError` 领域异常。
- 仅 `BusinessRuleError` 可被管线转换为 warning 并继续。
- `TypeError`、`LookupError`、`ValueError` 和其他未预期异常必须保留堆栈并重新抛给 Application 全局错误处理器。
- 未预期异常后禁止继续执行后续业务 handler。
- 测试必须调用真实 `_get_business_handlers()`，验证注册顺序与文档一致。
- 处理 PTB ConversationHandler 警告：回调入口采用与对话键一致的 tracking 配置，或移出不需要对话跟踪的 fallback；最终测试输出不得保留这三条警告。

## 6. 数据库迁移与 Schema Gate

### 6.1 Alembic 引导

仓库当前只有失效的 `alembic.ini`，没有脚本目录。本次建立完整 Alembic 环境：

1. 新建 async Alembic `env.py` 和版本目录。
2. 建立 `legacy_baseline` 修订，表示当前兼容 SQL 已覆盖的历史结构。
3. 首次启动检测不到版本表时，显式执行一次 legacy bootstrap，成功后 stamp baseline；日志必须记录执行和结果。
4. 后续启动只执行 `alembic upgrade head`，不再每次遍历全部兼容 SQL。
5. 可靠性状态字段、唯一约束和索引进入独立 revision，并提供 downgrade。

legacy bootstrap 是一次性迁移桥，不是静默 fallback；失败直接拒绝启动。

### 6.2 Schema Gate

Schema Gate 对模型要求的结构至少检查：

- 表和字段存在。
- 规范化后的 PostgreSQL 字段类型。
- nullable。
- server default。
- 外键目标与删除策略。
- 显式要求的普通索引和唯一约束。

差异全部列出后一次性失败，禁止只报告第一项。`BOT_ALLOW_SCHEMA_COMPAT` 不绕过迁移执行错误。

## 7. 操作与可观测性

每个领域页面统一展示：

- 待处理、处理中、重试中、永久失败、不确定和成功数量。
- 最近错误、首次/最近尝试时间、尝试次数和下次执行时间。
- “重试”“关闭”“查看原始任务”动作。
- 重放必须二次确认，并记录管理员 ID、原因和原 attempt ID。

Scheduler 状态增加每类结果计数、最近成功、最近失败和积压量。任务内部处理失败必须反映到 scheduler 健康状态，不能只写日志后返回成功。

## 8. 代码结构与质量门禁

### 8.1 拆分规则

对审计确认的 10 个超大文件和 237 个超长函数按责任拆分：

- handler 只做输入解析和流程编排。
- service 只做业务规则。
- repository 只做查询和持久化。
- presenter/view 只做文本与键盘。
- Telegram adapter 只做外部调用。

对外 handler、service 类名和路由入口保持稳定；删除拆分后的死代码和重复兼容路径。

### 8.2 自动门禁

新增可重复执行的质量检查，覆盖 `backend/` 生产 Python：

- 文件不超过 500 行。
- 函数不超过 50 行（空行除外）。
- 嵌套深度不超过 3。
- 位置参数不超过 3，超出时改用不可变配置对象。
- 圈复杂度不超过 10。
- 禁止新增未命名魔法数字。

新增 Ruff 和类型检查配置。门禁必须作用于全量目标目录，不使用仅检查“本次新增代码”的豁免方式。
